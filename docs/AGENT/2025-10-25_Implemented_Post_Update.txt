  Core Implementation Status

  - Inbox System - Receives activities with HTTP signature verification
  - Followers Management - Add/remove followers via Follow/Undo activities

  - HTTP Signatures - Complete signing/verification for all S2S communication
  - Activity Delivery - Auto-delivers Accept and Create activities to followers

> are post content in AP/AS meant to be clear TXT or can you add MD? 

‚óŹ Great question! Let me check the ActivityPub/ActivityStreams spec and common practices:

  ActivityStreams Spec:
  - The content field accepts HTML, not plain text or Markdown
  - The mediaType property indicates the format (defaults to text/html)

  Common Practice in the Fediverse:
  - Most servers (Mastodon, Pleroma, etc.) accept Markdown in the input
  - They convert it to HTML before storing in the content field
  - The content field in the ActivityPub JSON is always HTML
  - Plain text is also HTML-encoded (special chars escaped)

  Example from Mastodon:
  {
    "type": "Note",
    "content": "<p>This is <strong>bold</strong> and <em>italic</em> text.</p>",
    "contentMap": {
      "en": "<p>This is <strong>bold</strong> and <em>italic</em> text.</p>"
    }
  }

  For tinyFedi:
  Currently we just store whatever you pass as content directly. We should probably:

  Option A: Accept Markdown, convert to HTML before storing
  - Pros: User-friendly input, spec-compliant output
  - Cons: Need markdown library dependency

  Option B: Accept HTML directly
  - Pros: No conversion needed, full control
  - Cons: Less user-friendly

  Option C: Accept plain text, HTML-escape it
  - Pros: Simple, safe
  - Cons: No formatting support

  What would you prefer? I'd recommend Option A (Markdown ‚Üí HTML) as it's most user-friendly and matches what others do.

> ok. remind me what is the best way to add a link to the content and open it in a new tab 

‚óŹ In HTML (which is what ActivityPub content should contain), you'd use:

  <a href="https://example.com" target="_blank" rel="noopener noreferrer">link text</a>

  Attributes explained:
  - target="_blank" - Opens in new tab
  - rel="noopener noreferrer" - Security best practice (prevents new page from accessing window.opener)

  However, in the Fediverse context:

  Most ActivityPub implementations strip or ignore the target="_blank" attribute for security reasons. The client (Mastodon web UI, mobile apps, etc.)
  decides how to handle links, not the content creator.

  What happens in practice:
  - Mastodon web: Links open in new tab by default (client decides)
  - Mobile apps: Opens in-app browser or external browser (client decides)
  - The target attribute in the content is typically ignored

  Best practice for ActivityPub:
  Just use plain links without target:
  <a href="https://example.com" rel="noopener">link text</a>

  Or if you're using Markdown (and converting to HTML):
  [link text](https://example.com)

  The client application will handle how to open it. You can't really control whether it opens in a new tab from the server side in ActivityPub - that's a
   client-side decision.

  For tinyFedi specifically:
  Since we're likely linking to blog posts, a simple <a href="..."> is fine. The url field in the post already points to the canonical location, and
  users' clients will handle how to open links appropriately.
